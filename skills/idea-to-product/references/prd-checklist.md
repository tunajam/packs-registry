# PRD Quality Checklist

Use this checklist before finalizing any PRD.

## Problem Definition

- [ ] Problem is specific, not generic ("users struggle with X" not "we need a better Y")
- [ ] Problem is validated (found evidence of people complaining/paying to solve)
- [ ] Target user is clearly defined (persona, not "users")
- [ ] Root cause identified, not just symptoms

## Success Metrics

- [ ] Primary metric is quantified (not "improve engagement")
- [ ] Metrics are measurable with available tools
- [ ] Baseline measurement exists or can be established
- [ ] Target is ambitious but achievable
- [ ] Timeframe for measurement defined

## Scope

- [ ] Non-goals explicitly listed
- [ ] MVP is truly minimal (can ship in <2 weeks)
- [ ] Future features are documented but separated
- [ ] "DO NOT IMPLEMENT" section prevents scope creep
- [ ] Edge cases considered but not over-engineered

## User Stories

- [ ] Every story follows As/Want/So format
- [ ] Each story delivers user value (not technical tasks)
- [ ] Acceptance criteria are testable
- [ ] Criteria use specific values, not "appropriate" or "reasonable"
- [ ] Happy path AND error states covered

## Technical Requirements

- [ ] Requirements use active verbs (display, calculate, send)
- [ ] Each requirement is atomic (one thing)
- [ ] Requirements are verifiable (can test true/false)
- [ ] Performance targets specified where relevant
- [ ] Security/accessibility requirements included

## For AI Implementation

- [ ] No ambiguous terms (fast, appropriate, user-friendly)
- [ ] Specific values for all thresholds
- [ ] Error messages quoted exactly
- [ ] State transitions explicitly defined
- [ ] Protected code/schema clearly marked

## Stakeholder Alignment

- [ ] Engineering reviewed for feasibility
- [ ] Design reviewed for UX consistency
- [ ] Legal/security reviewed if applicable
- [ ] Dependencies on other teams identified
- [ ] Sign-off obtained (or scheduled)

## Documentation Quality

- [ ] Someone unfamiliar could understand the product
- [ ] No undefined acronyms or jargon
- [ ] Links to research/context included
- [ ] Open questions tracked
- [ ] Version/date clearly marked
